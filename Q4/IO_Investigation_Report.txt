2. FILE DESCRIPTOR OBSERVATIONS
----------------------------------------------------
Standard Linux File Descriptors:
  FD 0 → stdin  (standard input)   → pipe:[1138]
  FD 1 → stdout (standard output)  → pipe:[1139]
  FD 2 → stderr (standard error)   → pipe:[1140]

Command: ls -la /proc/$$/fd/0 /proc/$$/fd/1 /proc/$$/fd/2
Output:
  lr-x------ 1 root root 64  fd/0 -> pipe:[1138]   (read-only)
  l-wx------ 1 root root 64  fd/1 -> pipe:[1139]   (write-only)
  l-wx------ 1 root root 64  fd/2 -> pipe:[1140]   (write-only)

Observation:
File descriptors are non-negative integers that the kernel uses to track open
resources. FD 0 (stdin) is read-only while FD 1 (stdout) and FD 2 (stderr) are
write-only. In this environment they point to pipes rather than a terminal,
which is typical in non-interactive script execution. When a process opens a
file, it receives the next available FD (starting at 3).

Command: cat /home/claude/error_output.txt
Output:  "ls: cannot access '/nonexistent_directory': No such file or directory"
Result:  stderr (FD 2) was captured to file; terminal showed nothing

--- Combined stdout + stderr (2>&1) ---
Command: ls /home/claude/ /nonexistent_dir 2>&1 | tee /home/claude/combined_output.txt
Output:  Both the directory listing AND the error message appeared together
Result:  2>&1 merges stderr into stdout stream; tee splits to file AND screen

--- Input Redirection (<) ---
Command: wc -l < /home/claude/input_test.txt
Output:  3
Result:  File contents fed to wc via stdin redirection (3 lines counted)

--- /dev/null - Discarding Output ---
Command: ls /nonexistent 2>/dev/null
Output:  (nothing — error silently discarded)
Result:  /dev/null is a special device that discards all data written to it

--- tee Command ---
Command: echo "tee test output" | tee /home/claude/tee_output.txt
Output:  "tee test output" (displayed on screen AND written to file)
Result:  tee splits I/O stream: writes to both stdout and a file simultaneously

Observation:
Linux I/O redirection provides powerful control over where command input
comes from and where output goes. The distinction between stdout (1) and
stderr (2) allows programs to separate normal output from error messages.
Combining 2>&1 merges them into one stream, useful for logging all output.

Observation:

The most relevant limit for file I/O is 'nofiles' (1024 open files). The soft
limit (1024) can be raised by the process up to the hard limit (4096) using
'ulimit -n <value>'. The stack size limit (8192 KB) and max processes (15955)
also impact how applications interact with the OS.

5. EXPLANATION: HOW LINUX MANAGES FILE I/O
----------------------------------------------------

Linux File I/O Architecture:

  1. Virtual File System (VFS):
     Linux uses a VFS abstraction layer that allows different filesystems
     (ext4, tmpfs, procfs, etc.) to present a unified interface. All file
     operations go through the VFS layer regardless of the underlying storage.

  2. File Descriptor Table:
     Each process maintains a file descriptor table (visible in /proc/<PID>/fd/).
     When a file is opened with open(), the kernel returns an integer FD that
     the process uses for all subsequent read/write operations.

  3. I/O Streams and Redirection:
     The shell implements redirection by duplicating file descriptors before
     executing commands. For "> file", it opens the file and uses dup2() to
     replace FD 1 (stdout) with the file's FD. The process writes to "stdout"
     without knowing the output goes to a file.

  4. Pipes:
     Pipes (|) create an anonymous in-memory buffer. The kernel connects the
     stdout of the left command to the stdin of the right command via pipe(),
     enabling command chaining without temporary files.

  5. Buffered vs Direct I/O:
     By default, the kernel buffers file writes in the page cache for
     performance, flushing to disk periodically. Direct I/O bypasses this
     cache and writes directly to storage (used in databases).

  6. /proc Virtual Filesystem:
     /proc is a pseudo-filesystem where files are kernel data structures
     presented as regular files. Reading /proc/$$/fd reveals open descriptors;
     reading /proc/meminfo shows memory usage — no actual disk I/O occurs.
